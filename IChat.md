# 🚀 浏览器 AI Chat 插件：无摩擦上下文捕获方案 (Tech Spec)

## 📌 核心愿景
打破传统 AI 插件“手动复制粘贴”或“强制精准划词”的高操作成本，实现**“指哪打哪、所见即所得”**的 Zero-Friction (零摩擦) 交互体验。

---

## 🛠️ 核心架构与技术路线对比

### 方案 1：经典划词提取 (Text Selection)
*   **交互方式**：鼠标左键拖拽选中目标文本 -> 呼出/触发 AI。
*   **实现原理**：利用浏览器原生 API `window.getSelection().toString()`。
*   **优势 (Pros)**：用户意图 100% 明确，零误判；开发成本极低。
*   **劣势 (Cons)**：
    *   **操作成本高**：需要精准对齐起止位置，长文本跨页划词体验极差。
    *   **多模态/特殊场景失效**：对图片内的文字、Canvas 渲染的内容（如 Figma/飞书文档）、强行设置了 `user-select: none` 的防复制网站完全束手无策。

### 方案 2：基于坐标的 DOM 智能捕获 (Smart Hover)
*   **交互方式**：按住快捷键（如 `Alt`），鼠标悬停在目标段落上 -> 插件自动高亮预测区域 -> 松开按键触发 AI。
*   **实现原理**：
    1.  监听 `mousemove` 获取坐标 `(x, y)`。
    2.  调用 `document.elementFromPoint(x, y)` 获取最底层 DOM 节点。
    3.  **核心算法（向上冒泡）**：由于底层节点往往太细碎（如单个 `<span>`），需编写 Heuristics（启发式）算法：循环访问 `parentElement`，直到满足特定条件（如：文本长度 > 30 字符，或遇到 `<p>`, `<article>` 等语义化块级标签）才停止提取。
*   **优势 (Pros)**：极大地降低了物理操作成本，“指哪打哪”。
*   **劣势 (Cons)**：机器猜测的 DOM 范围可能与用户的真实心理预期不符（偏大或偏小）。
*   **🌟 UX 极致优化方案（必做）**：
    *   **视觉反馈**：必须在捕获的 DOM 元素上覆盖一层半透明色块或边框，实现“所见即所得”。
    *   **滚轮微调 (Scroll to Expand/Shrink)**：在按住快捷键悬停时，**向上滚动鼠标滚轮，高亮区域自动向父节点 (`parentElement`) 扩大；向下滚动则缩小到子节点**。将范围的最终决定权极其优雅地交还给用户。

### 方案 3：纯视觉范围框选 (Marquee Selection) *[进阶创新点]*
*   **交互方式**：按住快捷键，鼠标在屏幕上随意拖拽出一个矩形选框（类似截图软件），提取落入该选框内的所有文本。
*   **实现原理**：
    *   **路线 A (底层光标 API)**：记录拖拽起点 `(x1,y1)` 和终点 `(x2,y2)`，利用 `document.caretRangeFromPoint()` 获取两个点对应的文本节点，然后 `createRange()` 将它们连起来，一次性提取自然阅读流中的纯文本。
    *   **路线 B (DOM 碰撞检测)**：画出一个纯视觉的 `<div>` 选框，遍历网页 DOM 的 `getBoundingClientRect()`，利用 AABB（轴对齐边界框）数学碰撞算法，把“掉进框里”的 DOM 文本拼接起来。
*   **优势 (Pros)**：无视前端复杂的 Div 嵌套逻辑；完美解决网页“多栏排版（左右分列）”时传统提取方式容易串行的问题。

### 方案 4：基于 Vision LLM 的多模态截图降维打击
*   **交互方式**：触发快捷键 -> 屏幕变暗 -> 框选区域 -> 将截图直接发给 AI。
*   **实现原理**：调用 `chrome.tabs.captureVisibleTab` 截取当前屏幕，前端根据鼠标坐标裁剪图片，调用 GPT-4o Vision 或 Claude 3.5 Sonnet 等多模态大模型。
*   **优势 (Pros)**：降维打击。无视一切反爬虫、Canvas 渲染、图片海报、PDF文件，所见即能解。
*   **劣势 (Cons)**：Token 成本显著增加；API 响应延迟较高（通常比纯文本慢 1-3 秒）。

## 🎁 额外 Bonus 功能 (杀手级应用场景)

### Bonus 1：隐式上下文扩充 (Implicit Context) —— 解决“只选一个词，AI 乱解释”的痛点
*   **场景**：用户在阅读一篇关于量子力学的长文，遇到了一个专业名词“自旋”，用户用**方案 1（划词）**只选中了“自旋”两个字。
*   **痛点**：直接把“自旋”发给 AI，AI 可能会给出极其宽泛或偏离当前文章语境的解释。
*   **解决方案**：结合 **方案 1 + 方案 2**。
    *   插件通过 `window.getSelection().anchorNode` 获取用户选中单词所在的底层 DOM 节点。
    *   静默执行“向上冒泡算法”，抓取包含该单词的**整段话甚至整篇文章**。
    *   向 AI 组装隐式 Prompt：*“用户正在阅读以下内容：【此处插入静默抓取的整段上下文】，请在这个语境下，为用户解释单词【自旋】的意思。”*
*   **价值**：用户操作成本极低（只划一个词），但获得了拥有全局视野的精准回答。

### Bonus 2：智能降级与混合路由策略 (Fallback Routing)
*   **场景**：网页环境千奇百怪，没有任何单一方案能 100% work。
*   **解决方案**：在插件架构设计时，实现一套自动降级的“漏斗”机制：
    1.  **用户按下触发键时**：
        *   首选检测是否存在**划词选中 (Selection)**，如果有，直接走隐式上下文模式 (Bonus 1)。
        *   如果没有划词，执行**智能悬停检测 (Smart Hover / DOM)**。
    2.  **当 DOM 提取失败时（遇到图片/Canvas/防复制）**：
        *   插件自动或提示用户切换为 **多模态截图模式 (Vision LLM)** 作为最终兜底方案。
*   **价值**：保障了产品在任何极端前端环境下的高可用性 (SLA)。

## 📅 MVP (最小可行性产品) 落地路线图建议

*   **Phase 1 (核心爽点验证)**：
    *   实现**方案 2**（鼠标悬停 + 高亮边框 + 向上冒泡获取父节点文本）。跑通 Content Script 与 AI 面板的通信。
*   **Phase 2 (交互升华)**：
    *   加入**滚轮扩缩 (Scroll to Expand)** 功能，优化选区准确度。
    *   实现 **Bonus 1**（隐式上下文扩充）。
*   **Phase 3 (终极形态)**：
    *   集成**方案 4 (Vision 截图)** 作为兜底。
    *   挑战实现**方案 3 (纯视觉框选提取)**，彻底奠定该插件在 UX 上的领先地位。
